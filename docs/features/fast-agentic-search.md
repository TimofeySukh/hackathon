# Fast Search and Agentic Search

## Purpose

Fast search is the planned replacement for scan-heavy board search and the current fixed
smart-search pipeline. It should make ordinary search instant for names, notes, tags,
roles, events, and circles while keeping natural-language search as a transparent agentic
tool loop instead of a hardcoded AI router.

## Behavior

- Toolbar search returns deterministic results as the user types. It must not wait for an
  LLM when the query can be answered by indexed names, notes, tags, roles, events, or
  circle membership.
- Tags are first-class data, not disguised note text. Users may see them in a note-like
  area, but storage and search should treat them as canonical tag assignments loaded from
  a versioned catalog, not generated by the model.
- Signed-in smart search uses the same indexed search surface plus graph inspection tools.
  The model receives tool schemas, calls tools, observes their results, and repeats until
  it returns a final text answer with no tool calls.
- Tool calls and compact tool results are visible in the UI trace. The raw tool name,
  arguments, status, summary, and inspected result set should be available to the user.
- If the LLM provider is unavailable, deterministic search still works. Agentic search may
  fail with a clear unavailable state, but it must not degrade into a hidden prompt router.
- Search should stay responsive on large imported graphs. Heavy local indexing work runs
  outside the render path, ideally in a Web Worker.

## Non-Negotiable Constraints

- Do not implement hardcoded AI search routers, keyword-to-prompt branches, or
  query-specific shortcuts as the architecture.
- Do not classify a query with an LLM and then force it through one fixed hand-authored
  route. The model may choose among tools at runtime; the harness validates tools and
  returns observations.
- Do not place shared tags in `notes[]` as the long-term model. Notes remain free-form
  human/derived observations; tags are structured assignments.
- Do not read and scan the whole `user_graphs.graph` JSONB blob on the server hot path for
  every search once projection indexes exist.
- The agent loop must not rely on artificial prompt restrictions to control search shape.
  Real safety belongs in auth, scopes, tool permissions, validation, cancellation, and
  infrastructure budgets.

## Research Snapshot

Checked on 2026-07-08 against active repositories updated within the last four months:

- [OpenCode](https://github.com/sst/opencode): session processing stores tool calls as
  first-class parts with pending/running/completed/error states, emits events, filters
  tools through permissions, and uses loop/permission guards instead of a query router.
- [Pi Agent](https://github.com/earendil-works/pi): the harness loop repeatedly streams a
  model turn, validates and executes tool calls, appends tool results, and ends only when
  no tool calls remain. It has explicit lifecycle events for agent, turn, message, and
  tool execution.
- [GBrain](https://github.com/garrytan/gbrain): fast retrieval uses multiple independent
  search arms, Reciprocal Rank Fusion, source-aware SQL ranking, page-level max-pooling,
  typed entity recall, and fail-open relational recall. The ranking lives in search
  infrastructure, not in prompt branches.
- [Graphiti](https://github.com/getzep/graphiti): graph memory search combines BM25,
  cosine similarity, BFS graph expansion, RRF/MMR/cross-encoder reranking, typed filters,
  and per-group queues for ingestion. Search config chooses retrieval methods; tools
  expose node/fact search with filters.
- [Mem0](https://github.com/mem0ai/mem0): memory search validates scoped filters, runs
  semantic search plus keyword search, lemmatizes BM25 queries, boosts linked entities,
  supports metadata operators, and applies optional rerankers after over-fetching.

Reusable rules from the code:

- Build durable search projections and indexes first; use the LLM only after candidate
  retrieval is already cheap and bounded.
- Use several retrieval arms and fuse/rerank candidates instead of betting on one search
  method.
- Keep entity/tag/role/event filters structured and indexed.
- Treat ingestion and indexing as their own pipeline, with queues or revisioned
  projections when needed.
- Make tool execution observable. A model-selected tool path should be inspectable after
  the fact.

Code-level notes to preserve:

- OpenCode separates session processing from tools. Tool calls are stored as explicit
  session parts with lifecycle states, permission checks happen before execution, and
  repeated identical tool calls are treated as a loop risk. DataNode smart search should
  copy that separation: the loop records events; search tools only search.
- Pi Agent's loop has one simple invariant: a model turn may yield tool calls; every tool
  call gets a tool result; the next model turn receives those results; the agent stops only
  when the model returns text without tool calls. DataNode should implement this directly
  instead of another analyze/match/retry prompt pipeline.
- GBrain's retrieval code uses source-aware SQL ranking, page-level max-pooling, typed
  entity recall, and RRF. The important lesson for DataNode is to keep raw notes as source
  material while projecting people, tags, roles, events, and relationship facts into
  separate searchable rows.
- Graphiti's search recipes make retrieval strategy explicit: BM25, vector similarity,
  BFS graph expansion, and rerankers are composable options. DataNode should expose
  similarly narrow deterministic tools such as person search, tag search, role search,
  event search, person inspection, and neighbor expansion.
- Mem0 validates scope filters before retrieval, over-fetches from semantic search, adds
  keyword/BM25 and entity boosts, and only then applies optional rerankers. DataNode should
  keep owner/revision scope mandatory and apply semantic reranking only after bounded
  deterministic retrieval.

## Design

Follow [`../DESIGN_SYSTEM.md`](../DESIGN_SYSTEM.md).

- Surfaces / elevation used: the existing board search surface; optional inline trace
  rows below smart-search results.
- Components used: search field, icon buttons, chips for tags/roles, list rows for
  results, progress rows for tool calls, and disclosure controls for raw tool result
  payloads.
- Color roles used: existing Material 3 surface, primary, secondary, error, and outline
  roles only.
- Anything feature-specific: tool-call trace rows should be compact and scan-friendly,
  not marketing cards. Tags should render as stable chips or compact rows with source
  labels when needed.
- Known gaps vs. the Material 3 target: none until implementation.

## Code

Current files:

- `src/lib/search/graphSearch.ts` — local ranked scan over graph data.
- `supabase/functions/graph-api/graphSearch.ts` — duplicated server copy of local search
  ranking logic.
- `supabase/functions/graph-api/agentSearch.ts` — current fixed multi-pass smart-search
  pipeline. This should not be the long-term agentic search architecture.
- `supabase/functions/graph-api/index.ts` — `/v1/search/smart` endpoint.
- `scripts/datanode-cli.mjs` and `scripts/datanode-mcp.mjs` — CLI/MCP parity points for
  future search endpoints and tools.

Planned files / ownership:

- Supabase migration for search projections:
  `tag_definitions`, `person_tags`, `person_roles`, `person_events`, and
  `graph_search_documents`.
- Projection maintenance near graph writes in `supabase/functions/graph-api/index.ts`, or
  a small module imported by graph-write handlers.
- Shared deterministic ranking module used by both browser and Edge Function, avoiding
  the current duplicated `graphSearch.ts` drift.
- Browser-side indexed search module, potentially backed by a Web Worker, for anonymous
  and immediate signed-in search.
- Agent loop module for smart search, separate from retrieval tools. The loop owns model
  turns and event emission; tools own scoped deterministic operations.

## Implementation Plan

1. Baseline current lag and quality.
   - Add a repeatable local benchmark with large imported graphs and representative
     queries for name, company, role, tag, event, and vague natural-language searches.
   - Record latency targets for local search, server deterministic search, and smart
     search first-result/tool-trace visibility.

2. Move shared tags out of notes.
   - Add a versioned global tag catalog that is loaded from data, not generated by an
     LLM and not hardcoded into query logic.
   - Add person-tag assignments as structured graph data plus DB projections.
   - Migrate existing tag-like notes only through explicit migration rules with source
     preservation; keep ordinary notes as ordinary notes.

3. Add search projection tables and indexes.
   - Create `graph_search_documents` rows for people, notes, tags, roles, events, circles,
     and relationship facts.
   - Index exact ids, owner id, entity type, tag ids, role keys, event dates, circle ids,
     `tsvector`, and trigram text.
   - Store provenance and graph revision so stale projection rows can be detected and
     rebuilt.

4. Build deterministic fast search.
   - Query exact names/tags/roles, full-text, trigram, graph-neighborhood, and optional
     vector arms independently.
   - Fuse candidates with RRF or a similarly explainable scorer, then apply structured
     boosts for exact tag/role/name matches and circle proximity.
   - Return score details in debug mode so ranking failures can be diagnosed without
     changing prompts.

5. Extract roles and events as structured projections.
   - Derive role candidates from profile/headline/position notes and LinkedIn enrichment
     output.
   - Store event-like facts separately from prose notes with source, event kind, date,
     person id, and confidence/provenance.
   - Use deterministic filters to narrow role/event searches before any semantic rerank.

6. Replace fixed smart search with a real agent loop.
   - Define read-only tools such as `search_people`, `search_tags`, `search_roles`,
     `search_events`, `inspect_person`, and `expand_neighbors`.
   - On each turn, send the model the conversation plus available tool schemas. When it
     returns tool calls, validate and execute them, append tool results, emit trace
     events, and call the model again.
   - End only when the model returns final text without tool calls.
   - Keep safety in auth scopes, schema validation, result limits, user cancellation, and
     operational failsafes; do not encode search strategy as prompt-only restrictions.

7. Preserve interface parity.
   - Every new API search operation must have equivalent CLI commands, MCP tools, and
     public docs in `src/DocsPage.tsx`.
   - Tool results must use the existing structured envelope: `status`, `summary`, `data`,
     and `next_valid_actions`.

8. Validate before shipping.
   - Run unit tests for projection rebuilds and rank fusion.
   - Run load tests against large graphs and browser responsiveness tests.
   - Run golden-query evals against known imported data.
   - Inspect the final diff for forbidden patterns: hardcoded AI routers, prompt keyword
     branches, hidden whole-graph scans, and notes-as-tags persistence.

## Open Questions / TODO

- Decide whether first-class tags are stored canonically inside `GraphState`, relational
  tables, or both. The current product persists the graph as JSONB, so relational search
  tables may initially be projections.
- Decide who owns the global tag catalog and how catalog changes are versioned across
  existing user graphs.
- Decide whether embeddings are generated synchronously during graph writes, asynchronously
  after saves, or only for signed-in users that enable smart search.
- Define the exact latency budget after measuring real imported graphs.
